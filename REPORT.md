## 2. Реализация таблицы idempotency_keys

Таблица idempotency_keys
id - ключ записи вв бд
idempotency_key - ключ идемпотентности, который клиент передаёт в заголовке Idempotency-Key
request_method - HTTP-метод запроса
request_path - путь запроса (endpoint)
request_hash - хэш тела запроса, помогает понять клиент делает тот же самый запрос или нет 
status - состояние обработки ключа: processing, completed или failed
status_code - HTTP-код ответа - 200, 409
response_body - кэш тела ответа в формате JSONB
created_at - время создания записи
updated_at - время последнего обновления записи
expires_at — время истечения срока жизни записи ttl

Также есть ограничение на статусы(check) и защита от дублирования im_key, пути и метода запроса 
Индексы нужны для ускорения поиска по ключу и быстрой отчастки старых записей
## 3. Реализация middleware
_TODO: Опишите алгоритм middleware по шагам._

Минимум:
1. Чтение `Idempotency-Key`.
2. Проверка существующей записи.
3. Обработка кейса \"тот же key + другой payload\".
4. Сохранение результата первого запроса.
5. Возврат кэшированного ответа при повторе.

в целом тут идет обработка post запросов о платежах 
1. Читаем Idempotency key
2. Из тела запроса берем request_hash
3. Находим в таблице idempotency_keys запись о поптке оплаты
4. если запись найдена и request_hash отличается от сохранённого, возвращается 409;
   если статус равен completed, клиенту возвращается кэшированный ответ;
   если статус равен processing, возвращается сообщение о том, что запрос уже выполняется
5. Если записи нет то она создается
6. Сохраняется запрос
7. Всоедующий раз при этом запросе бужет возращен кэшированный ответ

запуск docker compose exec -T backend pytest app/tests/test_compare_idempotency_vs_for_update.py -v -s
for update защита от двойной оплаты на уровне бд, во избежании некорректного обновления данных 
Idempotency key используется для защиты от повторной отправки одного и того же API-запроса, чтобы оплата в данном случае не повторилась

при повторном запросе база данных не обновляется , оставляя запись только об одной оплате 
а оплата не проходит второй раз, так как запись об этом уже сохранена 

целосообразно использовать оба механихма совместно для лучшей и более надежной работы сервера
